🧠 核心设计决策

三个决策对抗三个失效模式

每条规则都可追溯到真实事故，每个决策都对抗至少一个系统性失效模式。

三层模型

L1 始终加载 · L2 激活时≤300行 · L3 按需读取。控制 context 膨胀，让有效指令不被挤出。

对抗：规则膨胀
🚪

阶段门控

每个阶段有独立的入口/出口条件，不满足不可跳过。门控是结构约束，不是建议。

对抗：规则遗忘
🔍

审计脚本+清单

自动化扫描 + 23项逐项确认。跨章节一致性只能靠脚本校验，目测=遗漏。

对抗：规则冲突
层级内容预算加载时机
L1触发词 + 跳过条件~100词始终
L2工作流、原则、速查表≤300行激活时
L3详细文档、schema、示例不限按需 read

🎯 创作工作流

8个阶段，每个都有入口和出口门控

从理解需求到发布——不满足出口条件，任何阶段都不可跳过。

1

理解需求

收集3+真实输入输出示例，明确Skill类型，确认不适用场景。

出口：输入输出示例≥3条
1.5

找实践案例

黄金案例3+，失败案例5+，有效-无效对比≥1组。

出口：案例清单就绪
2

规划架构

选意图模式、架构模式，列文件树，长流程定义progress.json。

出口：文件树+模式确定
3

写L1描述

触发词+跳过条件，双引号单行≤1024字符，无emoji无宣传语。

出口：description校验通过
4

写L2主文件

按需IRON LAW，阶段门控，确认门控，交付检查清单，≤300行。

出口：≤300行+有检查清单
5

写L3参考文档

每主题一文件，引用仅一跳，底部建索引。

出口：引用文件齐全+有索引
6

可验证性审查

逐条审查指令可yes/no判断。“写高质量代码”→“lint通过无error”。

出口：全部指令可验证
7

触发词优化

扫禁止词，中英文触发词5+种，测试10+10准确率≥85%。

出口：准确率≥85%
✅ 质量审计清单

23项检查，覆盖3大维度

自动化扫描+人工确认。“目测没问题”不再是交付标准。

📋 格式规范 9项

  • IRON LAW：业务专属约束
  • description：单行无emoji
  • name：小写连字符，与目录名一致
  • 行数≤300
  • 所有阶段编号+门控
  • 无硬编码凭据
  • 改动类有确认门控
  • 子Agent规范≤30行
  • 安全：无人员信息/保密信息

🏗️ 结构健康 10项

  • 新旧规则无矛盾
  • 同一内容未双源存储
  • 无僵尸文件
  • 无断链引用
  • 改名后路径已同步
  • 无用户态数据混入
  • 脚本无硬编码隐患
  • 无功能退化
  • 版本号与改动匹配
  • references/索引完整

📝 内容质量 7项

  • 跨章节一致性
  • 无冗余信息收集
  • 交互流程无矛盾
  • 文案无错别字/歧义
  • 降级链完整
  • 已知局限已披露
  • 循环有停滞检测

🚀 快速开始

三步开始使用

1

📦 安装

解压到你的 Skills 目录。

unzip mu-skill-creator.zip -d ~/.skills/
2

✨ 创建新 Skill

对 Agent 说“帮我创建一个新 Skill”，即可触发完整8阶段创作流程。

3

🔍 审计现有 Skill

运行审计脚本，对任意 Skill 执行23项质量检查。

bash scripts/skill-audit.sh <skill-name>

⚡ 别再目测了。
用工程方法。
