# mu-skill-creator

一个创建和审计 AI Agent 技能的工具——三层架构模型、八阶段门控工作流、23项质量清单，让"看起来没问题"变成可验证的工程实践。

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](https://opensource.org/licenses/MIT)

## 使用场景

- 🏗️ **从零创建新 Skill** — 8 阶段引导式工作流，从需求收集到发布一站式完成
- 🔍 **审计现有 Skill** — 23 项质量检查清单，捕捉反模式、结构问题和安全风险
- 📏 **强制 300 行预算** — 自动检测膨胀的 SKILL.md，建议 L3 拆分方案
- 🎯 **优化触发词** — 10+10 正反测试，目标准确率 ≥85%
- 🛡️ **发布前安全扫描** — 检测硬编码凭据、敏感系统 URL 和个人信息泄露
- 🧪 **运行 Eval 测试** — 有/无 Skill 对比，量化实际效果
- ⚠️ **停滞循环检测** — 内置 stale_count 机制，逃出无限迭代陷阱
- 📐 **验证指令可测试性** — 将主观的"写高质量代码"转化为客观的"lint 通过无 error"

## 核心亮点

### 三层架构模型 (L1/L2/L3)

通过分层加载控制 context 预算，从根本上对抗规则膨胀问题：

| 层级 | 内容 | 预算 | 加载时机 |
|------|------|------|---------|
| L1 | 触发词 + 使用/跳过条件 | ~100词 | 始终加载 |
| L2 | 工作流、原则、速查表 | ≤300行 | 激活时加载 |
| L3 | 详细文档、schema、示例 | 不限 | 按需读取 |

L1 是唯一触发机制，倾向"宁多触发别漏"——漏触发意味着 Skill 等于不存在，误触发可以纠正。L2 严守 300 行预算（官方建议 500，保守取 300 留余量），引用仅允许一级跳转。L3 按主题拆分，超 100 行加索引。

### 八阶段门控工作流

每个阶段有独立入口/出口条件，未满足不可跳过——门控是结构约束，不是"建议"：

| 阶段 | 名称 | 入口条件 | 出口条件 |
|------|------|---------|---------|
| 1 | 理解需求 | 收到创建/优化需求 | 输入输出示例清单≥3条 |
| 1.5 | 找实践案例 | 阶段1完成 | 黄金案例3+/失败案例5+ |
| 2 | 规划 | 阶段1.5完成 | 文件树+架构模式已确认 |
| 3 | 写 L1 | 阶段2完成 | 触发词+不适用场景 |
| 4 | 写 L2 | 阶段3完成 | ≤300行+有Checklist |
| 5 | 写 L3 | 阶段4完成 | 所有引用文件存在 |
| 6 | 可验证性审查 | 阶段5完成 | 全部指令可yes/no验证 |
| 6.5 | Eval测试 | 阶段6完成 | accuracy≥85% |
| 7 | 触发词优化 | 阶段6.5完成 | 无禁止词，准确率≥85% |
| 8 | 发布 | 阶段7+授权 | marketplace可搜到 |

### 23项质量检查清单

三大类覆盖从格式到内容的完整质量维度：

**格式规范（9项）**：IRON LAW 位置与内容、description 格式与触发词、intro 三段式一致性、命名规范、行数预算、阶段编号与门控、SSO 方案、Confirmation Gate、安全扫描。

**结构健康（10项）**：逻辑冲突与因果闭环、模板双源检测、僵尸文件清理、断链引用修复、路径同步、用户态数据隔离、硬编码检测、功能退化防护、版本号匹配、索引完整性。

**内容质量（7项）**：跨章节一致性、信息冗余消除、交互一致性、文案质量、降级链完整性、已知局限披露、停滞检测机制。

### 9条设计原则

分正向（怎么做好）和防御（怎么防坏）两个视角：

**正向设计**：状态外置优于上下文累积（进度写文件不写对话）；执行与评估分离（干活的不评判进度）；方向差异化优先于深度挖掘（多样性是逃出局部最优的路径）；每次迭代必须有可验证产出。

**防御设计**：规则必附事故来源（理解事故>死记禁止令）；职责隔离靠约束不靠自律；假设每层都会失败（确保有独立恢复路径）；停滞检测用量化指标；已知局限必须诚实披露。

### 20条反模式 (AP-1~AP-20)

每条标注根因事故和对应设计原则，从真实失败中提炼：

| # | 反模式 | 修复方案 |
|---|--------|---------|
| 1 | >250行 | 拆 references/，效果优先 |
| 2 | description写宣传语 | description=仅触发条件 |
| 3 | 阶段无编号/出口 | 加编号+完成定义 |
| 4 | 多跳引用A→B→C | 仅一跳 |
| 5 | 无完成门控 | 加验证步骤 |
| 6 | 不可验证指令 | 改yes/no可判断 |
| 7 | 无Confirmation Gate | 改动前加用户确认 |
| 8 | 无Pre-Delivery Checklist | 输出前加打勾清单 |
| 9 | IRON LAW是套话 | 改专属约束；无风险则删 |
| 10 | 无AP列表 | 加禁止行为清单 |
| 11 | 无子Agent执行规范 | 加≤30行最小规范 |
| 12 | 循环无终止条件 | 加max/超时退出 |
| 13 | 无数据量限制 | 加limit/截断/分页 |
| 14 | 冗余重复提示 | 同一指令只写一处 |
| 15 | 大文件无截断 | 加字数/行数上限 |
| 16 | SKILL.md留intro | intro只通过push管理 |
| 17 | Shell无shebang/set-euo | 加shebang+安全开关 |
| 18 | 重复造轮子 | bundle到scripts/复用 |
| 19 | 规则只有MUST没WHY | 附WHY解释意图 |
| 20 | IRON LAW照搬通用模板 | 必须含业务专属约束 |

按失效模式分类——规则遗忘：AP-3/5/8/17；规则冲突：AP-4/14/16/18；规则膨胀：AP-1/9/13/15/20。

### 停滞检测机制

适用于所有含循环/迭代的 Skill，通过量化信号而非主观判断来检测卡住：

| 信号 | 规则 |
|------|------|
| 单次迭代0新产出 | stale_count+1 |
| stale_count≥2 | 换结构约束（不是调参数） |
| stale_count≥4 | 上报人类 |
| 单阶段超过15轮或30分钟 | 强制换方向 |

核心理念："换结构不是调参数"——当任务在同一框架内反复停滞，决定性收益来自修正环境/结构约束本身。
